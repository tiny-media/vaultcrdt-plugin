## Befunde

## 1) HIGH — Initial-Sync überspringt Dateien anhand der VV, ohne den aktuellen lokalen Inhalt zu prüfen

**Betroffen:** `src/sync-initial.ts:166-185`

```ts
// 2. Overlapping docs — two-tier skip: VV+hash → full sync
if (cached && currentServerVV && vvEquals(currentServerVV, cached.vv)) {
  lastServerVV.set(file.path, currentServerVV);
  contentHashes.set(file.path, cached.contentHash);
  skippedVVMatch++;
  ...
  continue;
}
```

### Problem
Der Kommentar sagt „VV+hash“, aber implementiert ist praktisch nur „VV gleich => skip“.

Die Datei wird **nicht von Disk gelesen**, bevor sie übersprungen wird. Dadurch können lokale Änderungen übersehen werden, wenn:
- der Server unverändert ist,
- aber die Datei lokal außerhalb von Obsidian geändert wurde,
- oder Obsidian zwischenzeitlich geschlossen war.

### Konkretes Risiko
Ein Nutzer ändert lokal per externem Editor / Git / Syncthing eine Datei. Beim nächsten Start bleibt die Server-VV gleich, also wird die Datei übersprungen. Die lokale Änderung landet dann u. U. **gar nicht** im CRDT / auf dem Server.

### Warum das kritisch ist
Das ist kein theoretisches Cleanup-Thema, sondern ein **echter Korrektheitsfehler** im Sync.

### Empfehlung
- Beim Skip **zusätzlich den aktuellen Datei-Hash prüfen**.
- Wenn `cached.contentHash !== fnv1aHash(currentDiskContent)`, **nicht skippen**.
- Ergänze dafür gezielte Tests:
  - „server VV unverändert, lokale Datei extern geändert → darf nicht geskippt werden“
  - „server VV unverändert, lokale Datei unverändert → skip ist erlaubt“

## 2) HIGH — Delete-/Tombstone-Semantik erlaubt Wiederbelebung gelöschter Dateien

**Betroffen:**
- Server: `src/main.rs:16-27`
- Server: `src/handlers.rs:185-186`, `src/handlers.rs:234-235`
- Plugin: `src/push-handler.ts:42-49`
- Plugin: `src/sync-engine.ts:406-408`

### Beobachtung A — Tombstones laufen nach 7 Tagen ab

```rust
match db::expire_tombstones(&hourly_pool, 7).await {
```

### Beobachtung B — Jeder neue `sync_push` / `doc_create` entfernt den Tombstone automatisch

```rust
db::remove_tombstone(pool, vault_id, doc_uuid).await?;
```

### Beobachtung C — Lokales Delete räumt den persistierten CRDT-State nicht sauber weg

```ts
onFileDeleted(path: string): void {
  this.docs.remove(path);
  this.lastServerVV.delete(path);
  ...
}
```

und bei Remote-Delete:

```ts
private async onDocDeleted(docUuid: string): Promise<void> {
  this.docs.remove(docUuid);
  this.lastServerVV.delete(docUuid);
  ...
}
```

### Problem
Gelöschte Dateien können unter realistischen Offline-Szenarien zurückkommen:
- Gerät B ist länger offline.
- Gerät A löscht Datei X.
- Nach 7 Tagen verfällt der Tombstone.
- Gerät B verbindet sich wieder und pusht die alte Datei erneut.

Zusätzlich ist das Resurrection-Verhalten serverseitig sehr offen: ein `sync_push` oder `doc_create` räumt den Tombstone einfach weg.

### Risiko
- „Zombie-Dateien“
- Verwirrende Wiederauferstehung alter Inhalte
- Unerwartete Re-Konflikte nach längerer Offline-Zeit

### Empfehlung
- Tombstones **deutlich länger** halten oder konfigurierbar machen.
- Serverseitig strengere Re-Create-Regeln einführen:
  - entweder explizites „recreate“,
  - oder Resurrection nur erlauben, wenn der Client zuvor den Tombstone gesehen/acknowledged hat,
  - oder Delete-Generation / monotonen Delete-Marker speichern.
- Beim Delete lokal auch den persistierten State sauber entfernen (`removeAndClean` statt nur `remove`).
- Neue Tests:
  - „offline client reconnect after tombstone expiry“
  - „stale push after delete must not silently resurrect doc“

## 3) HIGH — Dateitypen und Pfade sind noch nicht sauber begrenzt

**Betroffen:**
- Plugin lokal: `src/main.ts:42-67`
- Plugin remote write: `src/sync-initial.ts:123-130`
- Plugin remote write: `src/editor-integration.ts:36-65`
- Server: `src/handlers.rs:54-90`, `src/handlers.rs:160-235`

### Problem 1 — lokal werden nicht nur Markdown-Dateien behandelt

`create` filtert auf `.md`, aber `modify` und `delete` nicht konsequent:

```ts
if (!(abstractFile instanceof TFile)) return;
...
const content = await this.app.vault.read(abstractFile);
```

und:

```ts
if (!(file instanceof TFile)) return;
...
this.syncEngine.onFileDeleted(file.path);
```

Damit können **nicht unterstützte Dateien** in den Sync-Pfad rutschen — besonders kritisch bei `delete`.

### Problem 2 — remote wird `doc_uuid` praktisch direkt als Vault-Pfad benutzt

```ts
await editor.writeToVault(uuid, doc.get_text());
```

und später:

```ts
await this.app.vault.create(filePath, content);
```

Serverseitig gibt es dafür aktuell keine harte Pfad-Validierung.

### Risiko
- Anhänge/Binaries können versehentlich per `doc_delete` auf anderen Clients gelöscht werden, obwohl ihr Inhalt gar nicht synchronisiert wird.
- Ein fehlerhafter oder bösartiger Client mit Vault-Credentials könnte Pfade unter `.obsidian/` oder sonstige unerwünschte Pfade schreiben/löschen lassen.

### Empfehlung
- Klare Allowlist definieren:
  - zunächst am sichersten: **nur `.md`**
  - optional später: definierte Text-Extensions (`.md`, `.txt`, `.csv`, …)
- Sowohl **clientseitig als auch serverseitig** validieren.
- Pfade unter `.obsidian/` standardmäßig blockieren.
- Keine Deletes/Broadcasts für nicht unterstützte Dateien erzeugen.

Commitete TS-Typen:


### Problem
Die im Plugin eingecheckten WASM-Artefakte stammen offenbar **nicht aus dem aktuellen Rust-Quellstand**.

### Empfehlung
- Plugin-WASM und Rust-Quelle **sofort synchronisieren**.
- Danach CI ergänzen, die sicherstellt:
  - generiertes WASM/JS/d.ts ist aktuell,
  - oder die Artefakte werden im Release-Prozess frisch gebaut.
- Rebuild-Anleitung einmal wirklich trocken testen.

## 5) HIGH — Secret-/Auth-Härtung reicht für Community-Release noch nicht

**Betroffen:**
- Schema: `/home/richard/projects/vaultcrdt-server/migrations/001_init.sql:3-7`
- DB-Verify: `/home/richard/projects/vaultcrdt-server/src/db.rs:66-105`
- Auth-Oracle: `/home/richard/projects/vaultcrdt-server/src/lib.rs:142-153`

### Problem
Vault-Passwörter (`api_key`) werden aktuell **im Klartext** gespeichert und per einfachem Stringvergleich geprüft.

```sql
api_key TEXT NOT NULL
```

```rust
let stored: String = r.get("api_key");
Ok(stored == api_key)
```

Außerdem verrät `/auth/verify` über unterschiedliche Fehlertexte, ob ein Vault existiert oder nicht:
- `Invalid API key`
- `Invalid admin token`

### Risiko
- Datenbank-Leak = sofortiger Secret-Leak aller Vaults
- brute-force / password spraying wird nicht erschwert
- Vault-Existenz lässt sich enumerieren

### Empfehlung
- `api_key` mit **Argon2id** oder `scrypt` hashen.
- Konstante Zeit beim Verify.
- Optional Rate-Limiting / Backoff für `/auth/verify`.
- Für Community-Release lieber generische Fehlermeldung statt klarer Existenz-Oracle.

## 6) MEDIUM — State-Datei-Encoding ist nicht kollisionsfrei

**Betroffen:** `src/state-storage.ts:20-23`

```ts
stateKey(filePath: string): string {
  return filePath.replace(/\.md$/, '').replace(/\//g, '__') + '.loro';
}
```

### Problem
Das Encoding ist nicht injektiv.

Beispiel:
- `a/__b.md` → `a____b.loro`
- `a__/b.md` → `a____b.loro`

### Risiko
Zwei unterschiedliche Dateien können denselben lokalen CRDT-State teilen/überschreiben.

### Empfehlung
- Statt `replace('/', '__')` eine kollisionsfreie Kodierung verwenden:
  - `encodeURIComponent(filePath)`
  - Base64url
  - Hash + Mapping-Datei
- Regression-Test für genau diesen Fall ergänzen.

## 7) MEDIUM — Surgical editor diff aktualisiert nur den ersten offenen Editor

**Betroffen:** `src/editor-integration.ts:87-145`

```ts
if (applied) return; // only apply to first matching editor
```

### Problem
Wenn dieselbe Datei in mehreren Leaves/Splits offen ist, wird der Diff nur auf den ersten Editor angewendet.

### Risiko
- Split-View zeigt inkonsistente Inhalte
- Nutzer sieht in einem Pane altes und im anderen neues Material
- kann Folgeprobleme mit weiteren Edits erzeugen

### Empfehlung
- Diff auf **alle** offenen Editoren derselben Datei anwenden.
- Dafür gezielte Tests mit 2 Leaves ergänzen.

**Betroffen:**
- Plugin: `src/sync-engine.ts:133-137`
- Server: `/home/richard/projects/vaultcrdt-server/src/ws.rs:100-105`

```ts
const url = `${this.wsUrl()}?token=${this.token ?? ''}&device=${device}&peer_id=${peerId}`;
```

und serverseitig:


### Risiko

### Empfehlung
Falls technisch möglich:

- single-use / handshake-token,
- oder alternativ ein anderes Upgrade-/Session-Modell.

Wenn Query-Tokens bleiben müssen, das in Deployment-Doku sehr klar als Log-Härtungs-Thema dokumentieren.
