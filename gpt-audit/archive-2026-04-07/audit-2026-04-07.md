## Befunde

## 1) HIGH — Offline-Deletes und Offline-Renames können beim Reconnect wieder auferstehen

**Betroffen:**
- Plugin: `src/push-handler.ts:7,42-55,88-94`
- Plugin: `src/sync-initial.ts:79-88,112-160,206-226`
- Plugin: `src/sync-engine.ts:95-102`

### Problem
Deletes werden offline nur in einem **in-memory Set** gehalten:

```ts
private pendingDeletes = new Set<string>();
```

Beim lokalen Delete wird die Datei lokal sofort entfernt, aber serverseitig nur dann gelöscht, wenn die WS-Verbindung offen ist:

```ts
if (this.isWsOpen()) {
  this.send({ type: 'doc_delete', ... });
} else {
  this.pendingDeletes.add(path);
}
```

Der Haken: In `runInitialSync()` werden **server-only docs zuerst heruntergeladen** und **erst danach** `flushPendingDeletes()` ausgeführt.

### Konkretes Fehlerszenario
1. Gerät ist offline.
2. Nutzer löscht `note.md`.
3. Plugin entfernt lokal Datei + `.loro`, merkt sich den Delete nur in `pendingDeletes`.
4. Reconnect / Neustart.
5. `request_doc_list` sieht `note.md` noch auf dem Server.
6. `initialSync()` behandelt `note.md` als **server-only** und lädt sie wieder herunter.
7. Erst danach wird `pendingDeletes` geflusht.

Damit ist der Delete-Pfad nicht robust. Je nach Timing passiert eines von zwei schlechten Dingen:

- **gleiches Session-Reconnect:** Datei wird lokal wieder angelegt, Delete geht danach zwar an den Server, aber der eigene Client bekommt sein `doc_deleted` nicht zurückgebroadcastet → lokal kann die Datei stehenbleiben.
- **Neustart vor Reconnect:** `pendingDeletes` ist komplett verloren, der Delete wird nie gesendet und die Datei kommt einfach zurück.

Dasselbe Problem betrifft Renames, weil `onFileRenamed()` intern `onFileDeleted(oldPath)` benutzt.

### Warum das wichtig ist
Deletes sind semantisch härter als normale Edits. Im aktuellen Stand sind **Edits offline robust**, **Deletes aber nicht**. Das ist für Freunde/Familie ein echter Vertrauensbruch: „Ich habe die Datei gelöscht“ darf nicht später in „sie ist wieder da“ enden.

### Empfehlung
- Offline-Deletes **persistent journalen** statt nur im RAM zu halten.
- Diese Queue **vor** dem Server-only-Download berücksichtigen.
- Bei `request_doc_list` server docs, die in der lokalen pending-delete-Queue stehen, **nicht** herunterladen.
- Tests ergänzen:
  - offline delete → reconnect → darf nicht redownloaded werden
  - offline delete → app restart → reconnect → Delete muss erhalten bleiben
  - offline rename → restart → old path darf nicht wiederkommen

## 2) HIGH — `doc_delete` umgeht den neuen Per-Document-Lock und reißt wieder ein Delete/Push-Race auf

**Betroffen:**
- Server: `src/handlers.rs:59-83`
- Server: `src/handlers.rs:160-214`
- Server: `src/handlers.rs:219-235`

### Problem
`sync_push` und `doc_create` werden korrekt unter `DocLocks` serialisiert:

```rust
let lock_key = DocLocks::lock_key(vault_id, &doc_uuid);
let lock = doc_locks.get(&lock_key);
let _guard = lock.lock().await;
```

`doc_delete` dagegen **nicht**:

```rust
msg::ClientMsg::DocDelete { doc_uuid, peer_id } => {
    db::delete_doc(pool, vault_id, &doc_uuid).await?;
    db::tombstone(pool, vault_id, &doc_uuid, &peer_id).await?;
```

### Warum der frühere Fix damit nur halb gilt
Der neue Tombstone-Schutz in `handle_sync_push()` / `handle_doc_create()` prüft zwar:

```rust
if db::is_tombstoned(...) { ... }
```

aber diese Prüfung ist nur dann stark, wenn Delete/Push/Create **wirklich serialisiert** sind.

Aktuell ist ein TOCTOU-Race möglich:
1. `sync_push` holt den Lock und prüft `is_tombstoned == false`
2. parallel kommt `doc_delete` rein und läuft **ohne Lock**
3. `doc_delete` löscht + tombstoned
4. `sync_push` schreibt danach trotzdem wieder `store_snapshot_with_vv(...)`

Dann kann der Server gleichzeitig
- ein Dokument in `documents`
- und einen Tombstone in `tombstones`

für denselben Pfad haben.

### Risiko
- Delete verliert gegen fast zeitgleichen Push
- inkonsistente `doc_list + tombstones` Zustände
- schwer debuggbares „manchmal wieder da, manchmal tombstoned“-Verhalten

### Empfehlung
- `DocDelete` unter **denselben `DocLocks`** stellen wie `sync_push` / `doc_create`.
- Regression-Tests ergänzen:
  - concurrent `doc_delete` vs `sync_push`
  - concurrent `doc_delete` vs `doc_create`
  - invariant: nach Delete darf nicht gleichzeitig aktives doc + Tombstone existieren

## 3) HIGH — Die Path-Policy ist noch nicht geschlossen: Initial-Sync und Rename-Kanten lassen verbotene Pfade durch bzw. löschen den alten Pfad nicht sauber

**Betroffen:**
- Plugin: `src/path-policy.ts:1-31`
- Plugin: `src/sync-initial.ts:48-57,79-88,167-218`
- Plugin: `src/main.ts:72-79`
- Plugin: `src/push-handler.ts:42-55`

### Problem A — Initial-Sync filtert lokale Dateien nicht mit `isSyncablePath()`
In `runInitialSync()` werden lokale Dateien so aufgebaut:

```ts
const localFiles = app.vault.getMarkdownFiles() as TFile[];
```

aber anschließend nur nach `tombstones` und `serverDocMap` gefiltert, **nicht** nach `isSyncablePath()`:

```ts
const overlappingFiles = localFiles.filter(
  (f) => !tombstoneSet.has(f.path) && serverDocMap.has(f.path)
);
const localOnlyFiles = localFiles.filter(
  (f) => !tombstoneSet.has(f.path) && !serverDocMap.has(f.path)
);
```

Damit ist die Policy, die sonst sauber eingeführt wurde, beim Startup wieder offen.

### Konkrete Folge
`.obsidian/*.md` oder `.trash/*.md` können beim Initial-Sync noch immer:
- als `localOnly` per `doc_create` hochgeladen werden
- als `overlapping` normal gemerged werden

Das ist genau die Klasse von Pfaden, die die zentrale Policy eigentlich ausschließen sollte.

### Problem B — Rename nur nach `newPath`, nicht nach Übergang alt→neu
Der Rename-Handler macht nur dann etwas, wenn **der neue Pfad** syncable ist:

```ts
if (file instanceof TFile && isSyncablePath(file.path)) {
  ...
  this.syncEngine.onFileRenamed(oldPath, file.path, content);
}
```

Dadurch gehen zwei Randfälle schief:

1. **syncable → unsyncable** (`note.md` → `.trash/note.md` oder `note.txt`)
   - kein `onFileRenamed()`
   - alter Server-Pfad wird **nicht tombstoned**
   - andere Geräte behalten die alte Datei

2. **unsyncable → syncable**
   - `onFileRenamed(oldPath, newPath, ...)` wird aufgerufen
   - `PushHandler` sendet `doc_delete` für `oldPath`, obwohl `oldPath` nie syncable war

### Empfehlung
- In `runInitialSync()` lokale Dateien schon beim Aufbau auf `isSyncablePath()` begrenzen.
- Rename nicht als „newPath ist gültig?“ behandeln, sondern als **Transition**:
  - alt syncable / neu syncable → rename
  - alt syncable / neu unsyncable → delete old only
  - alt unsyncable / neu syncable → create new only
  - alt unsyncable / neu unsyncable → ignore
- Tests ergänzen:
  - `.obsidian/foo.md` wird im initialSync nicht gepusht
  - rename `note.md -> note.txt` tombstoned den alten Pfad
  - rename `.obsidian/x.md -> note.md` erzeugt kein `doc_delete` für `.obsidian/x.md`

## 4) MEDIUM — Die TLS-/URL-Schutzlogik ist inkonsistent und im Setup sogar per String-Trick umgehbar

**Betroffen:**
- Plugin: `src/setup-modal.ts:120-123,137-145`
- Plugin: `src/settings.ts:103-115,304-380`
- Plugin: `src/sync-engine.ts:107-130`

### Problem A — Setup prüft „localhost“ nur per `includes()`
Die aktuelle Absicherung lautet:

```ts
if (/^(http|ws):\/\//i.test(this.serverUrl)
  && !this.serverUrl.includes('localhost')
  && !this.serverUrl.includes('127.0.0.1')) {
  ... reject ...
}
```

Das ist kein Host-Check, sondern nur ein Substring-Check. Beispiele, die **fälschlich erlaubt** werden:

- `http://localhost.evil.com`
- `http://example.com/?next=localhost`
- `http://127.0.0.1.evil`

### Problem B — Settings-UI umgeht den Schutz komplett
In `settings.ts` kann der Server einfach gespeichert werden; danach wird reconnectet. Es gibt dort **gar keine** TLS-Validierung. Anschließend nutzt `sync-engine.ts` die URL direkt für:
- `/auth/verify`
- WebSocket `/ws`
- Settings-Calls wie `/vault/peers` und `/debug/vault-stats`

### Risiko
Das ist kein Remote-Exploit „ohne Nutzeraktion“, aber eine reale Sicherheitslücke auf dem Weg zu Freunden/Familie:
- Tippfehler
- schlechte Anleitung
- copy-paste aus Blog/Issue/Chat
- lokales Test-Setup, das später versehentlich so bleibt

können dazu führen, dass Passwort und Notizinhalte über **Plain HTTP / WS** rausgehen.

### Empfehlung
- URL einmal zentral parsen (`new URL(...)`), nicht per String-Substring prüfen.
- Nur `http://localhost`, `http://127.0.0.1` und ggf. RFC1918-LAN-Ziele bewusst erlauben — wenn überhaupt.
- Dieselbe Validierung an **allen** Eingabepfaden erzwingen:
  - SetupModal
  - SettingsTab
  - SyncEngine-Start / Reconnect als letzte Schutzschicht
- Tests ergänzen für die Bypass-Beispiele oben.

## 5) HIGH — Das mitgelieferte `docker-compose.yml` startet mit öffentlich bekannten Secrets

**Betroffen:**
- Server: `docker-compose.yml:8-12`

### Problem
Das Repo enthält eine direkt lauffähige Compose-Datei mit:

```yaml
VAULTCRDT_JWT_SECRET: change-me-in-production
VAULTCRDT_ADMIN_TOKEN: change-me-in-production
```

Wenn jemand das Repo klont und „mal eben“ `docker compose up -d` macht, läuft ein Server mit **vorhersagbaren Secrets**.

### Risiko
Das ist für einen Internet-Service massiv:
- JWTs sind fälschbar
- Admin-Vault-Erstellung ist kompromittierbar
- die gesamte Auth-Grenze fällt praktisch weg

Gerade für Freunde/Familie und später GitHub/Community ist das ein klassischer Self-Host-Footgun.

### Empfehlung
- Compose fail-fast machen, z. B. mit `${VAULTCRDT_JWT_SECRET:?required}` und `${VAULTCRDT_ADMIN_TOKEN:?required}`.
- Beispielwerte nur in `.env.example`, nicht in einer direkt startbaren Compose-Datei.
- README so formulieren, dass `docker compose up` ohne gesetzte Secrets **gar nicht** funktioniert.

## 6) MEDIUM — Die öffentlichen READMEs sind nach Split + Auth-Härtung materiell veraltet

**Betroffen:**
- Plugin: `README.md:71`
- Server: `README.md:7-10,104-115,123-126`

### Beobachtung
Mehrere Aussagen sind inzwischen objektiv falsch:

- Server-README behauptet noch, das Server-Repo enthalte die WASM-/CRDT-Crates.
- Server-README behauptet noch, Vault-Passwörter würden im Klartext gespeichert.
- Plugin-README verweist für den WASM-Rebuild noch auf das Server-Repo statt auf die jetzt im Plugin-Repo liegenden Rust-Crates.

### Warum das mehr als Kosmetik ist
Für ein Projekt kurz vor Freunde/Familie / Community sind Docs Teil des Produkts.
Falsche Docs erzeugen:
- falsche Sicherheitsannahmen
- kaputte Build-Anleitungen
- unnötige Supportfälle
- Vertrauensverlust („was stimmt denn hier noch?“)

### Empfehlung
- Plugin-README und Server-README als eigenen kleinen Release-Block behandeln.
- Sicherheitsmodell, Repo-Layout und Build-Anleitung einmal vollständig auf den aktuellen Stand ziehen.
- Danach die Docs bei jedem strukturellen Umbau als first-class artefact behandeln, nicht als Nachgedanke.
